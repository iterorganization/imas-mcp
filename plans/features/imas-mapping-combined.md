# IMAS Mapping: Combined Schema Redesign + LLM Pipeline

Status: **Planning**
Priority: High — core deliverable
Supersedes: `mapping-redesign.md`, `imas-mapping-pipeline.md`, `imas-mappings.md`,
            `ids-mapping-architecture.md`

## Summary

Unified plan combining the mapping architecture redesign (schema renames, relationship
model, signal grouping unification) with the LLM-driven mapping pipeline (multi-step
discovery, CLI, prompts). These are complementary — the redesign establishes the graph
schema and naming that the pipeline produces into.

The redesign **must** come first: the LLM pipeline writes `MAPS_TO_IMAS` relationships
and `IMASMapping` nodes as defined by the redesign. Without the schema in place, the
pipeline has no target to write to.

Sequencing: six phases, each independently testable.

## Relationship Between the Two Plans

| Concern | mapping-redesign.md | imas-mapping-pipeline.md |
|---------|--------------------|-----------------------|
| **Problem** | Wrong schema: old names, dead nodes, two grouping systems | No automated way to create mappings |
| **Output** | Clean graph schema + renames + SignalGroup unification | CLI + multi-step LLM pipeline + prompts |
| **Dependencies** | None — pure schema/rename work | Needs redesign schema in place first |
| **Overlap** | Signal grouping design (what) | Signal grouping unification (when/where in pipeline) |
| **Risk if separate** | Pipeline targets wrong schema; renames break pipeline code | Pipeline can't start without clean schema |

The redesign is Phase 1–2 (foundation). The pipeline is Phase 3–6 (capability).

## MCP Tool Assessment

### IMAS Tools: Excellent

The recent refactoring (commits `9aa0e28`, `d05c0c9`) dramatically improved quality:
- `search_imas` returns 14+ fields per path (documentation, data_type, unit, ndim,
  node_type, physics_domain, COCOS, lifecycle, structure_reference, clusters,
  coordinates, introduced_in)
- Cross-references to facility signals, wiki mentions, code files
- BM25 + vector hybrid scoring with generic path filtering
- Cluster context for semantic grouping

These tools provide excellent context for the mapping pipeline's Step 1 (IDS
exploration) and Step 2 (field-level mapping).

### Signal Tool: Significant Quality Gap

`search_signals` is **missing most of the metadata** that has been carefully
generated by LLM enrichment.

are correct.

#### Missing Cross-References

The IMAS tool returns cross-references (facility signals, wiki mentions, code files)
via `_get_facility_crossrefs()`. The signal tool has **no equivalent**. The schema
defines `DOCUMENTS` relationships from `WikiChunk` → `FacilitySignal`, but the signal
tool doesn't traverse them.

#### Missing SignalGroup Context

The enrichment query traverses through `SignalGroup` to find IMAS mappings but
doesn't return the group's own metadata (`group_key`, `description`, `member_count`).

#### Remediation Required

The signal tool needs the same treatment the IMAS tool received in `9aa0e28`:

1. ~~**Fix the `unit` bug**~~ — **DONE** (schema, code, graph all use `unit` singular)
2. **Add missing properties** to the enrichment return: `keywords`, `aliases`,
   `sign_convention`, `cocos`, `analysis_code`, `tdi_function`, `tdi_quantity`
3. **Add wiki cross-references**: traverse `(wc:WikiChunk)-[:DOCUMENTS]->(s)`
4. **Add SignalGroup context**: return group `id`, `group_key`, `description`,
   `member_count` when the signal has a `MEMBER_OF` relationship
5. **Update the formatter** to render the new fields (metadata block, wiki refs,
   group context)
6. **Add `HAS_UNIT` traversal**: `(s)-[:HAS_UNIT]->(u:Unit)` for structured unit
   info (symbol, quantity type) — mirroring the IMAS tool pattern

This is a prerequisite for Phase 4 (mapping pipeline) — the pipeline's Step 1 
calls the search_signals underlying functions to gather signal context for the LLM.

## Phase 1: Schema Renames + Dead Weight Removal

**Goal**: Clean graph schema. All renames, deletions, and migration in one pass.
No backward compatibility.

### 1a. Node Renames

| Current | New | Rationale |
|---------|-----|-----------|
| `DataNode` | `SignalNode` | It's a signal source, not generic "data" |
| `IMASPath` | `IMASNode` | It's a node in the IMAS hierarchy |
| `IDSRecipe` | `IMASMapping` | Orchestrates IMAS mappings (old field-level IMASMapping node is gone) |
| `IDSRecipeStatus` | `IMASMappingStatus` | Follows node rename |
| `StructuralEpoch` | `SignalEpoch` | Represents signal validity ranges, not just structural changes |
| `DataNodePattern` | *(deleted)* | Replaced by `SignalGroup` in Phase 2 |

### 1b. Relationship Renames + Removals

| Current | Action |
|---------|--------|
| `FOLLOWS_PATTERN` | Rename to `MEMBER_OF` |
| `TARGET_PATH` | Delete (old IMASMapping node gone) |
| `SOURCE_PATH` | Delete (never created) |
| `INCLUDES_MAPPING` | Delete (IDSRecipe→IMASMapping gone) |
| `USES_SOURCE` | Delete (IDSRecipe→DataSource gone) |

### 1c. Dead Weight Deletion

- Delete `AgentSession` class + `AgentSessionStatus` enum (zero usage)
- Delete old `IMASMapping` class (field-mapping → relationship)
- Delete `MappingEvidence.created_by` slot (referenced AgentSession)
- Delete `DataNodePattern` class
- Remove `FacilitySignal.pattern_representative_id` and `pattern_template`
  (replaced by `MEMBER_OF` → `SignalGroup`)

### 1d. Remove Hardcoded Mapping Specs

Delete from `imas_codex/ids/graph_ops.py`:

Keep: `load_mapping()`, `load_sections()`, `load_field_mappings()`, `select_nodes()`,
`FieldMapping`, `Mapping` dataclasses (read-side utilities).

### 1e. Codebase Impact

~50 files, ~500+ occurrences. Key files listed in `mapping-redesign.md` §1e.

### 1f. Graph Migration Script

`scripts/migrate_v6.py`: label migrations, relationship migrations, vector index

### 1g. Verification

## Phase 2: SignalGroup Unification + New Relationships

**Goal**: Unified signal grouping schema. `MAPS_TO_IMAS` relationship model.
SignalGroup replaces both DataNodePattern and FacilitySignal pattern mechanics.

### 2a. SignalGroup Schema

New class in `facility.yaml` (see `mapping-redesign.md` §2a for full YAML).
Key fields: `id`, `facility_id`, `group_key`, `member_count`, `representative_id`,
`description`, `keywords`, `physics_domain`, `status` (discovered → enriched),
`embedding`.

### 2b. New Relationships

| Relationship | From → To | Purpose |
|---|---|---|
| `MAPS_TO_IMAS` | SignalGroup/SignalNode → IMASNode | Field-level mapping with transform properties |
| `POPULATES` | IMASMapping → IMASNode (STRUCT_ARRAY) | Assembly section target with config |
| `USES_SIGNAL_GROUP` | IMASMapping → SignalGroup | Assembly source composition |
| `MEMBER_OF` | SignalNode/FacilitySignal → SignalGroup | Group membership |
| `HAS_EVIDENCE` | SignalGroup → MappingEvidence | Evidence for mapping |

### 2c. Common Grouping Layer

Create `imas_codex/discovery/base/grouping.py`:
- **Detection stays separate** — regex patterns (FacilitySignal) and tree structure
  (MDSplus) are fundamentally different algorithms
- **Unified creation**: `create_signal_group(facility, group_key, member_ids, representative_id)`
- **Unified claiming**: `claim_signal_groups(facility, claim_token, batch_size)`
- **Unified propagation**: `propagate_group_enrichment(group_id, enrichment_dict, member_label)`
- **Terminal state**: groups must reach `enriched` during signal discovery

Refactor `parallel.py` and `mdsplus/graph_ops.py` to use common layer.

### 2d. Discovery CLI Refactor

Rename functions in both pipelines per `mapping-redesign.md` §2d. Signal grouping
runs at discovery time, not mapping time.

### 2e. IMASMapping Node (was IDSRecipe)

Assembly orchestration node. See `mapping-redesign.md` §3a for full schema.
Key: `id` = `{facility}:{ids_name}`, links via `USES_SIGNAL_GROUP` and `POPULATES`.

### 2f. Assembly Refactor

Graph-led traversal replaces JSON config. See `mapping-redesign.md` §3c for
the `IDSAssembler._assemble_from_graph()` implementation.

### 2g. Verification

- SignalGroup nodes can be created and queried
- `MAPS_TO_IMAS` relationships carry transform properties
- Assembly produces identical IDS output via new graph traversal
- Both grouping systems (regex + tree) produce SignalGroup nodes

## Phase 3: Signal Tool Quality Remediation

**Goal**: Bring `search_signals` to the same quality level as `search_imas`.
This is a prerequisite for the mapping pipeline — the LLM needs rich signal context.

### 3a. Enrich `_enrich_signals()` Cypher

**Unit property**: Already correct — `s.unit AS unit` matches the schema property
`unit` (singular). No change needed (fixed in prior refactor).

Add missing properties to the RETURN clause:
```cypher
s.keywords AS keywords, s.aliases AS aliases,
s.sign_convention AS sign_convention, s.cocos AS cocos,
s.analysis_code AS analysis_code,
s.tdi_function AS tdi_function, s.tdi_quantity AS tdi_quantity,
s.enrichment_source AS enrichment_source
```

Add `HAS_UNIT` traversal for structured unit info (canonical unit symbol from the
Unit node, complementing the `s.unit` string property):
```cypher
OPTIONAL MATCH (s)-[:HAS_UNIT]->(su:Unit)
```
Return `su.symbol AS unit_symbol`.

### 3b. Add SignalGroup Context

In the enrichment query, return group metadata when available:
```cypher
OPTIONAL MATCH (s)-[:MEMBER_OF]->(grp:SignalGroup)
...
RETURN ...,
       grp.id AS signal_group_id,
       grp.group_key AS signal_group_key,
       grp.description AS signal_group_description,
       grp.member_count AS signal_group_member_count
```

### 3c. Add Wiki Cross-References

Mirror the IMAS tool's `_get_facility_crossrefs()` pattern:
```cypher
OPTIONAL MATCH (wc:WikiChunk)-[:DOCUMENTS]->(s)
...
collect(DISTINCT wc.section) AS wiki_mentions
```

### 3d. Update Formatter

- Render `keywords` as comma-separated list
- Render `aliases` as comma-separated list
- Render `sign_convention` and `cocos` in metadata block
- Render `analysis_code` when present
- Render signal group context (group key, member count)
- Render wiki cross-references
- Render `unit_symbol` from HAS_UNIT traversal alongside `unit` property

**Note**: The formatter already uses `sig.get("unit")` which is correct after
the schema standardization to `unit` (singular). No fix needed.

### 3e. Verification

- `search_signals` returns `unit` for signals that have them (already working)
- Keywords, aliases, sign convention, COCOS visible in output
- Wiki cross-references appear when DOCUMENTS relationships exist
- Signal group context shown for grouped signals
- `unit_symbol` from HAS_UNIT traversal shown when available

### 4c. Verification

## Phase 5: Mapping Pipeline — Prompts + Orchestrator + CLI

**Goal**: Wire the 3-step LLM pipeline end to end.

### 5a. Prompt Templates

Create in `imas_codex/agentic/prompts/mapping/`:

| Template | Context Injected | Task |
|----------|-----------------|------|
| `exploration.md` | Signal groups, DD subtree, semantic search results | Assign groups to IMAS sections |
| `field_mapping.md` | Section assignment, detailed fields, unit analysis, COCOS | Generate field mappings with transforms |
| `validation.md` | Proposed mappings, validation results, existing mappings | Review, correct, finalize |

Register schema providers in `prompt_loader.py`:
- `imas_dd_subtree` — formatted DD tree
- `signal_groups` — signal group summary
- `imas_field_detail` — detailed field info
- `unit_analysis` — pint compatibility results
- `cocos_analysis` — sign-flip paths

### 5b. Pipeline Orchestrator

Create `imas_codex/ids/mapping.py`:

```python
async def generate_mapping(
    facility: str,
    ids_name: str,
    *,
    model: str | None = None,
    dd_version: str | None = None,
) -> MappingResult:
    """Generate IMAS mapping via multi-step LLM pipeline.
    
    Step 0: Gather signal groups + DD context
    Step 1: LLM assigns groups to IMAS sections
    Step 2: For each section, LLM generates field mappings
    Step 3: LLM validates and finalizes
    Persist: Write to graph
    """
```

Each step uses `call_llm_structured()` with step-specific response model and
injected context. Cost tracked per step.

### 5c. CLI

> **CLI section superseded by `cli-unification.md`.** Map commands are now under

### 5d. Escalation Persistence

### 5e. Verification

## Open Questions

1. **Shot-to-epoch resolution**: `ids export` takes `--epoch VERSION` but no shot
   number resolution exists. Likely: epoch-agnostic mappings, epoch selected at
   export time.

2. **Model tier**: Start with `language` (gemini-3-flash) for cost. Escalate to
   `agent` (claude-sonnet) if quality insufficient.

3. **Batch vs per-group in Step 2**: Start with per-section (efficient). Split to
   per-group if quality drops.

4. **Signal tool remediation timing**: Phase 3 can run in parallel with Phase 2
   since they touch different files. But Phase 4 depends on both.
