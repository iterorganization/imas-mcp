# 24: COCOS Sign Convention Propagation to Standard Names

## Problem Statement

Standard names generated from IMAS DD paths currently receive only a documentation
hint when the source path is COCOS-dependent (`⚠️ include sign convention`). The
COCOS transformation type (e.g., `psi_like`, `ip_like`) and the DD version's COCOS
convention (11 for DD 3.x, 17 for DD 4.x) are not persisted on StandardName nodes,
not enforced during generation, and not injected into prompts with enough physics
context for the LLM to produce accurate, evidence-backed sign convention statements.

This plan propagates COCOS metadata through the standard names pipeline — following
the same authoritative-injection pattern already used for units — and enriches the
compose prompt with version-aware COCOS sign convention physics so the LLM can
generate precise "Sign convention: Positive when ..." documentation grounded in the
DD's declared convention.

## Design Principles

1. **Catalog-wide COCOS uniformity.** All standard names in a catalog share one
   COCOS version (currently COCOS 17, sourced from DD 4.x). The extraction DD
   version is recorded on each StandardName (`extraction_dd_version`) providing
   an auditable provenance path to the COCOS convention used.

2. **`cocos_transformation_type` is per-name.** Not all standard names are
   COCOS-dependent. Those that are carry the transformation type (`psi_like`,
   `ip_like`, `b0_like`, etc.) as a property, analogous to how `canonical_units`
   distinguishes `Wb` from `A`. This field is authoritative from the DD —
   injected post-LLM, never generated by the LLM.

3. **Follow the unit pattern.** Units flow:
   `DD graph → extract → enrich → prompt (read-only context) → post-process inject → persist (property + relationship)`.
   COCOS transformation type follows the identical pipeline stages.

4. **Graph-sourced, not hardcoded.** COCOS parameters are queried from the graph
   at extract time via `DDVersion → HAS_COCOS → COCOS`. Sign guidance prose is
   rendered from a small template YAML (`llm/config/cocos_sign_guidance.yaml`)
   filled with COCOS node properties — no per-version hardcoded dicts in Python.
   The pipeline fails closed for DD versions with no COCOS node link.

5. **Facility signal implications.** A uniform COCOS convention means facility
   signals stored in a different COCOS cannot receive a standard name without
   explicit COCOS conversion. This is intentional — it forces COCOS awareness
   at the mapping boundary rather than hiding convention mismatches inside names.
   Consumers must not confuse "standard name is COCOS-sensitive" with "linked
   signal is already normalized to COCOS 17".

## Evidence Base

### Current COCOS Data in the Graph

| Node Type | Property | Example | Source |
|-----------|----------|---------|--------|
| DDVersion | `cocos` (int) | `17` (DD 4.x), `11` (DD 3.x) | DD XML `<cocos>` element |
| DDVersion | `has_cocos` → COCOS node | `(DDVersion {id: "4.1.0"})-[:HAS_COCOS]->(COCOS {id: 17})` | build_dd.py |
| DDVersion | `cocos_labeled_fields` (int) | ~680 (3.42.2), ~250 (4.x backfilled) | DD XML attribute count |
| IMASNode | `cocos_label_transformation` | `psi_like`, `ip_like`, `b0_like` | DD XML attribute / backfill |
| IMASNode | `cocos_transformation_expression` | `"- {psi_like}"`, `"{ip_like} * {b0_like}"` | DD XML |
| IMASNode | `cocos_label_source` | `xml`, `inferred_forward`, `inferred_expression` | build_dd.py inference |
| COCOS (16 singletons) | `sigma_bp`, `e_bp`, `sigma_r_phi_z`, `sigma_rho_theta_phi` | COCOS 17: σ_Bp=-1, e_Bp=1, σ_RφZ=+1, σ_ρθφ=+1 | Sauter & Medvedev Table I |
| COCOS (derived booleans) | `psi_increasing_outward`, `phi_increasing_ccw`, `theta_increasing_cw`, `sign_q`, `sign_dp_dpsi` | COCOS 17: ψ decreasing outward, φ CCW, θ CW, q>0 | Computed from Sauter params |

### COCOS Transformation Types (COCOSLabelTransformation enum)

554 total COCOS-dependent fields across the DD, grouped into 16 transformation
types. The physics-relevant types for standard names:

| Type | Description | Field Count | Key Fields |
|------|-------------|-------------|------------|
| `psi_like` | Poloidal flux ψ and related | ~50 across 25 IDSs | `profiles_1d/psi`, `psi_axis`, `psi_boundary` |
| `ip_like` | Plasma current and related | ~30 across many IDSs | `global_quantities/ip`, `j_parallel`, `j_phi` |
| `b0_like` | Toroidal field and related | ~87 across 22 IDSs | `vacuum_toroidal_field/b0`, `profiles_1d/f`, `profiles_1d/phi` |
| `q_like` | Safety factor and related | ~8 across 6 IDSs | `profiles_1d/q`, `q_95`, `q_axis` |
| `dodpsi_like` | Derivatives w.r.t. ψ | ~4 in equilibrium | `dpressure_dpsi`, `dvolume_dpsi`, `f_df_dpsi` |
| `tor_angle_like` | Toroidal angle φ | ~50 across 4 IDSs | `position` angle fields |
| `pol_angle_like` | Poloidal angle θ | ~4 across 2 IDSs | `theta`, `poloidal_angle` |

Grid/tensor types (`grid_type_dim*_like`, `grid_type_tensor_*_like`) are structural
coordinates unlikely to generate standalone standard names — they will flow through
the pipeline but produce `metadata`-kind names or be classified as `skip`.

### Current Pipeline Gaps

1. **No COCOS property on StandardName schema.** Cannot query "which standard
   names are COCOS-dependent?" or "which names are psi_like?".

2. **No COCOS relationship in graph.** No `HAS_COCOS_TRANSFORMATION` link from
   StandardName to a transformation type or from StandardName to the COCOS node.

3. **Prompt provides label only, not physics.** The compose prompt shows
   `COCOS transformation: psi_like` but does not explain what COCOS 17 means
   for psi_like fields (ψ decreases from axis to edge for positive Ip). The
   LLM must guess the sign convention from training data alone.

4. **No post-process injection.** Unlike units, the COCOS label is not injected
   after LLM output. The LLM could hallucinate a wrong transformation type.

5. **No batch-level COCOS consistency check.** A batch could mix psi_like and
   non-COCOS paths without flagging the difference.

### COCOS 17 Sign Convention Reference

From Sauter & Medvedev (2013) and the DD 4.x specification:

| Parameter | COCOS 17 Convention |
|-----------|-------------------|
| σ_Bp (poloidal flux sign) | -1 |
| e_Bp (flux normalization) | 1 (full ψ, not ψ/2π) |
| σ_RφZ (cylindrical handedness) | +1 (right-handed) |
| σ_ρθφ (poloidal handedness) | +1 (right-handed) |

**Physical consequences for standard name documentation:**

| Transformation Type | COCOS 17 Sign Convention Statement |
|--------------------|-----------------------------------|
| `psi_like` | "Sign convention: Positive when the poloidal magnetic flux ψ is evaluated with the surface normal in the direction of increasing vertical coordinate Z. In COCOS 17, ψ decreases from the magnetic axis to the plasma boundary for standard positive-Ip tokamak operation." |
| `ip_like` | "Sign convention: Positive when the plasma current flows in the direction of increasing toroidal angle φ (counter-clockwise when viewed from above in COCOS 17)." |
| `b0_like` | "Sign convention: Positive when the vacuum toroidal magnetic field points in the direction of increasing toroidal angle φ (counter-clockwise when viewed from above in COCOS 17)." |
| `q_like` | "Sign convention: The safety factor q is positive when the toroidal and poloidal field components have the same helicity sense. In standard tokamak operation with co-current B₀, q > 0." |
| `dodpsi_like` | "Sign convention: The sign follows from the ψ convention (COCOS 17). Since ψ decreases outward for positive Ip, dX/dψ has the opposite sign to the radial gradient dX/dr." |
| `tor_angle_like` | "Sign convention: The toroidal angle φ increases counter-clockwise when viewed from above (right-handed cylindrical system, COCOS 17)." |
| `pol_angle_like` | "Sign convention: The poloidal angle θ increases counter-clockwise in the (R, Z) plane (right-handed system, COCOS 17)." |

**Add `extraction_dd_version` relationship:**

```yaml

  range: DDVersion
  annotations:
    relationship_type: EXTRACTED_FROM_DD_VERSION
```

This creates both a property (fast WHERE filter) and a relationship (graph
traversal) following the dual-model convention. The relationship enables:
```cypher
MATCH (sn:StandardName)-[:EXTRACTED_FROM_DD_VERSION]->(dv:DDVersion)-[:HAS_COCOS]->(c:COCOS)
RETURN sn.id, c.id AS cocos_version, c.sigma_bp
```

**Rationale for `cocos_transformation_type` as string (not enum range):**
- Referencing `COCOSLabelTransformation` from `imas_dd.yaml` would create a
  cross-schema dependency. `standard_name.yaml` is designed to be DD-independent.
- Moving the enum to `common.yaml` is an option for a future schema cleanup, but
  not required for this feature.
- A string property enables simple `WHERE sn.cocos_transformation_type = 'psi_like'`
  filtering without JOIN overhead.
- The existing COCOS semantic clusters provide graph traversal for discovery.

### Phase 3: Prompt Enhancement (Graph-Sourced Templates)

**Key principle: No hardcoded COCOS prose.** Sign guidance is rendered from a
template YAML file filled with properties from the COCOS graph node. The COCOS
node already stores both the raw Sauter parameters (`sigma_bp`, `e_bp`,
`sigma_r_phi_z`, `sigma_rho_theta_phi`) and derived boolean properties
(`psi_increasing_outward`, `phi_increasing_ccw`, `theta_increasing_cw`,
`sign_q`, `sign_dp_dpsi`) that express the physics in direction-based terms.

**New file: `imas_codex/llm/config/cocos_sign_guidance.yaml`**

One template per transformation label. Variables are filled from COCOS node
properties at runtime — the same template works for any COCOS version:

```yaml
# Sign guidance templates for COCOS-dependent standard name generation.
# Variables are filled from COCOS graph node properties (queried at extract time).
# No per-COCOS-version hardcoding — works for any COCOS with a graph node.

labels:
  psi_like:
    concept: "poloidal magnetic flux ψ"
    guidance: >-
      The poloidal magnetic flux ψ {psi_direction} from the magnetic axis to
      the plasma boundary for standard positive-Ip operation (σ_Bp = {sigma_bp}).
      State the sign relative to the surface normal direction (increasing Z).
    variables:
      psi_direction:
        from: psi_increasing_outward
        true: "increases"
        false: "decreases"

  ip_like:
    concept: "plasma current"
    guidance: >-
      Plasma current Ip is positive when flowing in the direction of
      {phi_direction} (σ_Bp = {sigma_bp}, σ_RφZ = {sigma_r_phi_z}).
      Current density j follows the same convention.
    variables:
      phi_direction:
        from: phi_increasing_ccw
        true: "increasing toroidal angle φ (counter-clockwise viewed from above)"
        false: "decreasing toroidal angle φ (clockwise viewed from above)"

  b0_like:
    concept: "vacuum toroidal field"
    guidance: >-
      The vacuum toroidal field B₀ is positive when pointing in the direction
      of {phi_direction} (σ_RφZ = {sigma_r_phi_z}). The toroidal flux
      function F = RBφ follows the same sign.
    variables:
      phi_direction:
        from: phi_increasing_ccw
        true: "increasing φ (counter-clockwise viewed from above)"
        false: "decreasing φ (clockwise viewed from above)"

  q_like:
    concept: "safety factor"
    guidance: >-
      The safety factor q = dΦ/dψ where Φ is toroidal flux. With
      σ_ρθφ = {sigma_rho_theta_phi}, q is {q_sign} for standard
      co-current operation.
    variables:
      q_sign:
        from: sign_q
        1: "positive"
        -1: "negative"

  dodpsi_like:
    concept: "ψ derivatives"
    guidance: >-
      Derivatives with respect to ψ inherit the ψ sign (σ_Bp = {sigma_bp}).
      Since ψ {psi_direction} outward, dX/dψ has {derivative_relation} sign
      to the radial gradient dX/dr for positive Ip.
    variables:
      psi_direction:
        from: psi_increasing_outward
        true: "increases"
        false: "decreases"
      derivative_relation:
        from: psi_increasing_outward
        true: "the same"
        false: "opposite"

  tor_angle_like:
    concept: "toroidal angle"
    guidance: >-
      The toroidal angle φ increases {phi_direction}
      (σ_RφZ = {sigma_r_phi_z}, {handedness} cylindrical system).
    variables:
      phi_direction:
        from: phi_increasing_ccw
        true: "counter-clockwise when viewed from above"
        false: "clockwise when viewed from above"
      handedness:
        from: sigma_r_phi_z
        1: "right-handed"
        -1: "left-handed"

  pol_angle_like:
    concept: "poloidal angle"
    guidance: >-
      The poloidal angle θ increases {theta_direction} in the (R, Z)
      poloidal plane (σ_ρθφ = {sigma_rho_theta_phi}, {handedness}).
    variables:
      theta_direction:
        from: theta_increasing_cw
        true: "clockwise"
        false: "counter-clockwise"
      handedness:
        from: sigma_rho_theta_phi
        1: "right-handed"
        -1: "left-handed"

generic_fallback: >-
  This quantity transforms under COCOS convention changes. Consult the
  transformation expression to determine the sign convention.
```

**New function in `context.py` — render guidance from template + COCOS node:**

```python
from imas_codex.llm.prompt_loader import load_prompt_config

def render_cocos_guidance(label: str, cocos_params: dict) -> str:
    """Render sign guidance for a transformation label using COCOS node properties.

    Args:
        label: COCOS transformation label (e.g., 'psi_like')
        cocos_params: Properties dict from the COCOS graph node
            (sigma_bp, psi_increasing_outward, phi_increasing_ccw, etc.)

    Returns:
        Rendered guidance string for the LLM prompt.
    """
    config = load_prompt_config("cocos_sign_guidance")
    label_config = config.get("labels", {}).get(label)
    if not label_config:
        return config.get("generic_fallback", "")

    guidance = label_config["guidance"]
    # Substitute raw Sauter parameters directly
    for param in ("sigma_bp", "sigma_r_phi_z", "sigma_rho_theta_phi", "e_bp"):
        guidance = guidance.replace(f"{{{param}}}", str(cocos_params.get(param, "?")))
    # Resolve template variables from COCOS node boolean/sign properties
    for var_name, var_spec in label_config.get("variables", {}).items():
        source_prop = var_spec["from"]
        source_val = cocos_params.get(source_prop)
        replacement = var_spec.get(str(source_val).lower() if isinstance(source_val, bool) else str(source_val), f"[unknown {source_prop}]")
        guidance = guidance.replace(f"{{{var_name}}}", replacement)
    return guidance
```

This function is called during prompt rendering — no version-specific dict, no
maintenance burden when a new COCOS version appears in the graph. Adding COCOS 3
support (if needed) requires only creating the COCOS 3 node in the graph.

Grid/tensor types (`grid_type_*_like`, `one_like`) fall through to the generic
fallback automatically since they have no template entry.

The pipeline fails closed: if `cocos_params` is None (no COCOS node linked to
the DD version), raise a clear error rather than generating names with no context.

**New relationship — EXTRACTED_FROM_DD_VERSION:**

```cypher
-- Create EXTRACTED_FROM_DD_VERSION relationships
-- Use MATCH (not MERGE) — DDVersion nodes already exist from DD build.
-- MERGE could create stub nodes with no HAS_COCOS, breaking provenance.
UNWIND $batch AS b
MATCH (sn:StandardName {id: b.id})
WHERE b.extraction_dd_version IS NOT NULL
WITH sn, b
MATCH (dv:DDVersion {id: b.extraction_dd_version})
MERGE (sn)-[:EXTRACTED_FROM_DD_VERSION]->(dv)
```

**Fail-safe:** If the `MATCH (dv:DDVersion ...)` returns no rows (version not in
graph), the relationship is silently skipped. The property
`sn.extraction_dd_version` still records the version string for audit. Log a
warning if any batch items have `extraction_dd_version` but no relationship was
created — this indicates a graph data issue.

This enables the full provenance chain:
```cypher
MATCH (sn:StandardName)-[:EXTRACTED_FROM_DD_VERSION]->(dv:DDVersion)-[:HAS_COCOS]->(c:COCOS)
RETURN sn.id, dv.id AS dd_version, c.id AS cocos, c.sigma_bp
```

### Phase 6: Validation & Consistency Checks

**New: Name-level COCOS conflict detection in `consolidation.py`:**

Add a check that when the same standard name appears in multiple batches (cross-IDS
dedup), the `cocos_transformation_type` values are consistent. This catches cases
where `_merge_duplicates()` would silently merge candidates with different COCOS types:

```python
def _check_cocos_conflict(existing: dict, incoming: dict) -> str | None:
    """Check for COCOS transformation type conflict between duplicate names."""
    existing_type = existing.get("cocos_transformation_type")
    incoming_type = incoming.get("cocos_transformation_type")
    if existing_type and incoming_type and existing_type != incoming_type:
        return (
            f"COCOS conflict for '{existing['id']}': "
            f"existing={existing_type}, incoming={incoming_type}"
        )
    return None
```

Integrate into the existing conflict detection at the same stage as unit/kind

**New: Pre-compose batch COCOS validation:**

Before composition, verify that all items in a batch with a `cocos_label` share
the same transformation type. This is a belt-and-suspenders check — the
(cluster × unit) grouping should already separate types, but mixed-type clusters
could slip through:

```python
def _check_batch_cocos_consistency(items: list[dict]) -> list[str]:
    """Verify all COCOS-dependent paths in a batch share the same type."""
    cocos_types = {
        item.get("cocos_label")
        for item in items
        if item.get("cocos_label")
    }
    if len(cocos_types) > 1:
        return [f"Mixed COCOS types in batch: {sorted(cocos_types)}"]
    return []
```

Call this in compose_worker before prompt rendering. If a batch contains mixed
COCOS types, **split the batch** into sub-batches grouped by `cocos_label` (each
sub-batch gets its own LLM call with consistent COCOS context). Do not send
mixed-type batches to the LLM — this defeats the sign guidance injection.


## What This Plan Does NOT Do

1. **Does not store COCOS version (int) on StandardName.** The COCOS version is
   traceable via `extraction_dd_version` → `DDVersion.cocos`. Storing `cocos: 17`
   on every name would be redundant and create drift risk.

2. **Does not create a new node type.** No `COCOSTransformationType` node.
   The transformation type is a string property matching the existing
   `COCOSLabelTransformation` enum. The existing COCOS semantic clusters
   provide the graph-traversal equivalent.

3. **Does not handle facility signal COCOS conversion.** When `--source signals`
   is used, signals from facilities using non-COCOS-17 conventions will NOT
   receive COCOS metadata. This is correct — COCOS conversion belongs at the
   mapping boundary (IMASMapping), not in the standard name. Consumers must
   not confuse "this standard name concept is COCOS-sensitive" with "this linked
   signal is already normalized to COCOS 17".

4. **Does not add COCOS to the ISN grammar.** COCOS transformation type is
   metadata about the quantity, not a grammar segment. It influences
   documentation content, not the name string itself.

5. **Does not add a new review scoring dimension.** COCOS compliance is covered
   by existing `documentation` and `compliance` dimensions plus tightened I2.5.
   Adding a 7th dimension would require updating normalization (`/120` → `/140`),
   calibration, prompts, benchmarks, and persistence — disproportionate effort.

6. **Does not move `COCOSLabelTransformation` to `common.yaml`.** The enum stays
   in `imas_dd.yaml`; StandardName uses `range: string`. A future schema cleanup
   could share the enum, but it's not required for this feature.

7. **Does not hardcode per-COCOS prose in Python.** Sign guidance is rendered from
   `llm/config/cocos_sign_guidance.yaml` templates filled with COCOS graph node
   properties. Adding support for a new COCOS convention requires only that the
   COCOS node exists in the graph — no code or config changes needed.

## Testing Strategy

1. **Extract data-flow test**: Verify `cocos_label` is present on batch items
   AFTER `_ENRICHED_QUERY` runs (before batch context is built). Confirm
   `extraction_dd_version` and `cocos_version` are resolved.

2. **Batch context test**: Verify `build_batch_context()` includes COCOS
   transformation type and convention when items have `cocos_label`.

3. **Pre-compose consistency test**: Verify `_check_batch_cocos_consistency()`
   flags mixed COCOS types within a single batch.

4. **Consolidation conflict test**: Verify `_check_cocos_conflict()` detects
   same-name candidates with different `cocos_transformation_type` during merge.

5. **Post-process injection test**: Verify `cocos_transformation_type` is
   injected from source item's `cocos_label` for psi_like paths and is None
   for non-COCOS paths.

6. **Graph persistence test**: Write StandardName with `cocos_transformation_type`
   and `extraction_dd_version`, read back, verify both property and
   `EXTRACTED_FROM_DD_VERSION` relationship exist.

7. **Catalog round-trip test**: Run `sn publish` → verify YAML contains
   `cocos_transformation_type` → run `sn import` → verify graph value matches.

8. **Template rendering test**: Verify `render_cocos_guidance("psi_like", cocos_17_params)`
   produces guidance mentioning "decreases" (σ_Bp=-1), and the same call with
   COCOS 11 params produces "increases" (σ_Bp=+1). Verify unknown labels return
   the generic fallback. Verify missing COCOS node properties produce `[unknown ...]`
   markers rather than KeyError.

9. **Version-aware prompt test**: Render `compose_dd.md` with COCOS 17 context
   and verify sign guidance appears. Render with COCOS 11 and verify different
   guidance appears (driven by different `cocos_params` from the graph).

10. **Prompt regression / quality test**: Run `sn generate --ids equilibrium --dry-run`
    and verify COCOS-dependent names (psi, ip, q, b0) have `cocos_transformation_type`
    set. Compare sign convention paragraph quality with and without enhanced context.

11. **Graph query validation**: After generation, verify:
    ```cypher
    MATCH (sn:StandardName) WHERE sn.cocos_transformation_type IS NOT NULL
    ```
    returns expected count (~50-100 names from equilibrium + core_profiles).
    Verify provenance chain:
