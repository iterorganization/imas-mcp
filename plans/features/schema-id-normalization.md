# Schema ID Normalization

**Date:** 2026-03-09
**Status:** Planned
**Type:** Breaking refactor
**Scope:** All LinkML schemas, graph client, Cypher queries, test fixtures, graph data

## Problem

Nine node types use a non-`id` field as their LinkML `identifier: true`. This causes two categories of bugs:

### Bug 1: Silent relationship creation failures

`create_nodes()` hardcodes `{id: item.slot_name}` when matching the **target** node of a relationship:

```python
# client.py line ~603
MATCH (t:{rel.to_class} {{id: item.{rel.slot_name}}})
```

When the target class's identifier is `name`, `path`, `symbol`, or `hostname`, this MATCH finds nothing. The relationship is silently not created. **24 schema-defined relationships are affected.**

### Bug 2: Cognitive overhead and query inconsistency

Developers must remember which identifier field each class uses. The codebase already shows widespread confusion:

- `graph_ops.py` mixes `{id: ...}` and `{path: ...}` for DataNode
- Test fixtures use `{id: ...}` for Unit and IDS (wrong per schema)
- Some code creates both `id` and `path` on DataNode as a workaround

## Decision

**Normalize all identifier fields to `id`.** No backward-compatible aliases. Clean break.

Every class gets `id` as `identifier: true`. The natural key (`path`, `name`, `symbol`, `hostname`) becomes a regular required property. The `id` value follows the existing composite key convention: `facility:natural_key`.

## Affected Classes

| Class | Current Identifier | Has `id` Slot? | New `id` Value Format | Natural Key Kept As |
|-------|-------------------|----------------|----------------------|---------------------|
| DataNode | `path` | Yes (non-identifier) | Keep existing `id` (e.g., `tcv:results:\RESULTS::PSI`) | `path` (required) |
| DataSource | `name` | Yes (non-identifier) | Keep existing `id` (e.g., `tcv:results`) | `name` (required) |
| MDSplusServer | `hostname` | No | `facility:hostname` (e.g., `tcv:tcvdata`) | `hostname` (required) |
| DiscoveryRoot | `path` | No | `facility:path` (e.g., `tcv:/home/codes`) | `path` (required) |
| Diagnostic | `name` | No | `facility:name` (e.g., `tcv:thomson_scattering`) | `name` (required) |
| AnalysisCode | `name` | No | `facility:name` (e.g., `tcv:LIUQE`) | `name` (required) |
| IDS | `name` | No | Same as `name` (e.g., `equilibrium`) | `name` (required) |
| Unit | `symbol` | No | Same as `symbol` (e.g., `Wb`) | `symbol` (required) |
| IdentifierSchema | `name` | No | Same as `name` (e.g., `coordinate_identifier`) | `name` (required) |

### ID format rationale

- **Facility-scoped classes** (DataNode, DataSource, MDSplusServer, DiscoveryRoot, Diagnostic, AnalysisCode): Use `facility:natural_key` to ensure global uniqueness.
- **Global classes** (IDS, Unit, IdentifierSchema): `id` = natural key. These are shared across facilities and already globally unique.
- **DataNode, DataSource**: Already have an `id` property set in practice with the composite format. Only the schema `identifier: true` annotation moves.

## Implementation Plan

### Phase 1: Schema Changes

Modify the three LinkML schema files. For each affected class:

1. Add `id` slot with `identifier: true` if it doesn't exist
2. Remove `identifier: true` from the natural key slot
3. Keep the natural key as `required: true`
4. Add `description` to the `id` slot documenting the composite key format

### Phase 2: Graph Client Simplification

#### client.py — `create_nodes()`

1. Remove the `id_field` parameter (default was `"id"`, now always `"id"`)
2. Remove `id_field` from the node MERGE query — hardcode `id`:
```python
# BEFORE
node_query = f"""
    UNWIND $batch AS item
    MERGE (n:{label} {{{id_field}: item.{id_field}}})
    SET n += item
"""

# AFTER
node_query = f"""
    UNWIND $batch AS item
    MERGE (n:{label} {{id: item.id}})
    SET n += item
"""
```
3. The relationship target query is now correct by definition:
```python
# This was the bug — now works because all identifiers are `id`
MATCH (t:{rel.to_class} {{id: item.{rel.slot_name}}})
```

#### client.py — `create_relationship()`

Remove `from_id_field` and `to_id_field` parameters. Hardcode `id`:
```python
# BEFORE
def create_relationship(self, from_label, from_id, to_label, to_id, rel_type,
                         from_id_field="id", to_id_field="id", props=None):

# AFTER
def create_relationship(self, from_label, from_id, to_label, to_id, rel_type,
                         props=None):
```

#### server.py — `add_to_graph()`

Remove `id_field` lookup — no longer needed:
```python
# BEFORE
id_field = schema.get_identifier(node_type)
...
result = client.create_nodes(label=node_type, items=valid_items, id_field=id_field, ...)

# AFTER
result = client.create_nodes(label=node_type, items=valid_items, ...)
```

### Phase 6: Graph Data Migration

The existing graph has nodes where the identifier property is the old natural key, and some nodes lack an `id` property entirely. A Cypher migration script sets `id` on all affected nodes.

#### Pre-migration: Backup

```bash
uv run imas-codex graph backup
```

#### Migration Script

Run in Cypher shell (`uv run imas-codex graph shell`) or via `python()` MCP:

```cypher
// 1. DataNode — id already exists on most nodes, verify all have it
//    For nodes missing id, construct from facility_id + data_source_name + path
MATCH (n:DataNode)
WHERE n.id IS NULL
SET n.id = n.facility_id + ':' + n.data_source_name + ':' + n.path;

// 2. DataSource — id already exists on some nodes
//    For nodes missing id, construct from facility_id + name  
MATCH (n:DataSource)
WHERE n.id IS NULL
SET n.id = n.facility_id + ':' + n.name;

// 3. Unit — add id = symbol
MATCH (n:Unit)
SET n.id = n.symbol;

// 4. IDS — add id = name
MATCH (n:IDS)
SET n.id = n.name;

// 5. IdentifierSchema — add id = name
MATCH (n:IdentifierSchema)
SET n.id = n.name;

// 6. Diagnostic — add id = facility_id:name
MATCH (n:Diagnostic)
SET n.id = n.facility_id + ':' + n.name;

// 7. AnalysisCode — add id = facility_id:name
MATCH (n:AnalysisCode)
SET n.id = n.facility_id + ':' + n.name;

// 8. MDSplusServer — add id = facility_id:hostname
MATCH (n:MDSplusServer)
SET n.id = n.facility_id + ':' + n.hostname;

// 9. DiscoveryRoot — add id = facility_id:path
//    (DiscoveryRoot is private-only, rarely in graph, but handle for completeness)
MATCH (n:DiscoveryRoot)
SET n.id = n.facility_id + ':' + n.path;
```

#### Post-migration: Re-create missing relationships

After migration, re-run ingestion pipelines that previously produced silent relationship failures. The relationships that were broken (listed above) will now work correctly through `create_nodes()`.

Alternatively, run targeted Cypher to create the most important missing relationships from existing property data:

```cypher
// DataNode -> DataSource (IN_DATA_SOURCE) from property data
MATCH (dn:DataNode)
WHERE dn.data_source_name IS NOT NULL
MATCH (ds:DataSource {name: dn.data_source_name, facility_id: dn.facility_id})
WHERE NOT EXISTS((dn)-[:IN_DATA_SOURCE]->(ds))
MERGE (dn)-[:IN_DATA_SOURCE]->(ds);

// DataNode -> Unit (HAS_UNIT) from property data

WHERE dn.unit IS NOT NULL
MATCH (u:Unit {id: dn.unit})
```

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| DataNode nodes missing `id` property | Low | Migration script sets it | Verify with COUNT query post-migration |
| DataNode `id` format inconsistent across ingestion paths | Medium | Some use `facility:tree:path`, others use `path` directly | Audit and normalize in migration script |
| Existing relationships created via workaround Cypher survive | None | Existing relationships are fine — they already exist. Only new ones were broken | N/A |
| Graph backup fails | Low | Data loss during constraint drop | Always backup before migration |

## Not In Scope

- Backward-compatible aliases (explicitly rejected)
- `id_field` parameter retention for "flexibility"
- Gradual migration with dual-identifier support
- Changes to classes that already use `id` as identifier (FacilityPath, IMASPath, etc.)
